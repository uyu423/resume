## 1. 현재 이력서 구조 분석

### 1.1 현재 섹션 구성

| 순서 | 섹션 | 역할 | 데이터 소스 |
|------|------|------|-------------|
| 1 | Profile | 프로필 사진, 이름, 연락처, 공지 | `payload/profile.ts` |
| 2 | Introduce | 장문 자기소개 + 서명 + 최종 업데이트일 | `payload/introduce.ts` |
| 3 | Skill | 카테고리별 기술 스택 (레벨 1~3) | `payload/skill.ts` |
| 4 | Experience | 회사/조직별 경력 (직책, 기간, 설명, 키워드) | `payload/experience.ts` |
| 5 | Project | 주요 프로젝트 (기간, 소속, 설명) | `payload/project.ts` |
| 6 | OpenSource | 오픈소스 기여 (링크, 뱃지) | `payload/openSource.ts` |
| 7 | Presentation | 발표/강연 이력 | `payload/presentation.ts` |
| 8 | Article | 수상/기고 이력 | `payload/article.ts` |
| 9 | Education | 학력 | `payload/education.ts` |
| 10 | Etc | 기타 활동 | `payload/etc.ts` |
| 11 | Footer | 버전, GitHub 링크 | `payload/footer.ts` |

### 1.2 강점 (자기PR 관점)

1. **데이터 기반 아키텍처**: `payload/` 파일만 수정하면 이력서가 완성되는 구조. 콘텐츠와 프레젠테이션이 분리되어 유지보수가 탁월함
2. **총 경력 기간 자동 계산**: Experience 섹션에서 전체 경력 기간을 자동으로 계산하여 Badge로 표시 — 채용담당자가 한눈에 연차를 파악 가능
3. **재직 중 뱃지**: `endedAt`이 없으면 자동으로 "재직 중" 뱃지 표시 — 현재 재직 상태의 시각적 강조
4. **Skill Keywords**: 경력 항목마다 사용 기술을 Badge로 표시 — 기술 매칭에 용이
5. **OpenGraph/SEO 기본 지원**: `next-seo` + OpenGraph 메타태그로 소셜 공유 시 미리보기 제공
6. **섹션별 disable 플래그**: 상황에 따라 섹션 노출/비노출 유연하게 제어 가능
7. **정적 HTML 배포**: GitHub Pages로 빠른 로딩, 무료 호스팅, 커스텀 도메인 지원
8. **오픈소스 섹션의 외부 뱃지 연동**: GitHub Stars, npm Downloads 등 실시간 소셜 증거(social proof) 표시

### 1.3 약점 (자기PR 관점)

#### A. 콘텐츠 구조적 문제

| 문제 | 설명 | 영향도 |
|------|------|--------|
| **히어로 영역 부재** | 첫 화면에서 "이 사람이 누구인지" 3초 안에 파악할 수 없음 | 🔴 높음 |
| **브랜드 스테이트먼트 없음** | "나는 OOO를 하는 OOO입니다" 같은 한 줄 정의가 없음 | 🔴 높음 |
| **자기소개 가독성 저하** | Introduce 섹션이 긴 영문 단락으로 구성 — 스캔이 어렵고 핵심이 묻힘 | 🟠 중간 |
| **성과 중심이 아닌 업무 나열** | Experience/Project 설명이 "what I did" 형태. "what impact I made"가 부족 | 🔴 높음 |
| **Article 섹션 네이밍 혼란** | 실제로는 수상/매체 노출 내용인데 "Article"이라는 이름이 기술 블로그 글로 오해될 수 있음 | 🟡 낮음 |
| **추천사/평가 섹션 없음** | 제3자의 검증(동료 추천, 리뷰 등)이 완전히 부재 | 🟠 중간 |

#### B. 시각/인터랙션 문제

| 문제 | 설명 | 영향도 |
|------|------|--------|
| **단조로운 비주얼** | 파란색 제목 + 회색 텍스트만으로 시각적 계층 구조가 약함 | 🟠 중간 |
| **스킬 레벨 표현 빈약** | 숫자 1~3에 의존, 시각적 프로그레스 바나 차트 없음 | 🟠 중간 |
| **스크롤 내비게이션 없음** | 긴 페이지에서 원하는 섹션으로 빠르게 이동할 수 없음 | 🟠 중간 |
| **인터랙티브 요소 부재** | 정적 텍스트 나열뿐, 사용자 참여를 유도하는 요소 없음 | 🟡 낮음 |
| **다크 모드 미지원** | 개발자 이력서인데 다크 모드가 없으면 기술 감각이 떨어져 보임 | 🟡 낮음 |
| **인쇄/PDF 최적화 없음** | 채용 프로세스에서 종이 출력이나 PDF 첨부가 필요할 때 대응 불가 | 🟠 중간 |

#### C. SEO/공유 최적화 문제

| 문제 | 설명 | 영향도 |
|------|------|--------|
| **구조화 데이터 없음** | JSON-LD (Person, ProfilePage) 미적용 — 구글 검색 결과에서 리치 스니펫 불가 | 🟠 중간 |
| **시맨틱 HTML 부족** | `<section>`, `<article>`, `<nav>` 등 시맨틱 태그 미사용 | 🟡 낮음 |
| **meta description 일반적** | "This is a simple web resume sample" — 검색 결과에서 클릭을 유도하지 못함 | 🟠 중간 |
| **접근성(a11y) 미검증** | alt 텍스트, ARIA 라벨 등 체계적 접근성 미지원 | 🟡 낮음 |

## 2. 채용담당자/면접관 심리 기반 콘텐츠 전략

### 2.1 채용담당자의 이력서 열람 패턴

연구에 따르면 채용담당자는 이력서를 평균 **6~7초** 스캔합니다. 이 시간 동안의 시선 흐름:

```
[1단계] 이름/직함 → [2단계] 현재 회사/직책 → [3단계] 핵심 기술 → [4단계] 경력 연차 → [5단계] 차별점
```

#### 현재 이력서의 문제

- **1단계**: 이름은 보이지만 "무엇을 하는 사람인지" 직함/한줄 소개가 없음
- **2단계**: Experience까지 스크롤해야 현재 직장을 알 수 있음 (Profile 바로 아래가 Introduce 장문)
- **3단계**: Skill 섹션이 3번째이긴 하지만, 카테고리별 텍스트 나열이라 빠른 파악 어려움
- **4단계**: 총 경력은 Experience 제목 옆 Badge로만 표시
- **5단계**: 차별화 요소가 페이지 하단(OpenSource, Presentation)에 매몰

### 2.2 심리학 기반 콘텐츠 재배치 전략

#### 전략 1: "3초 규칙" — 프로필 히어로 강화

**채용담당자의 심리**: "이 사람이 우리 팀에 필요한 사람인가?"를 3초 안에 판단

**제안**: Profile 섹션에 다음 요소 추가
- **한 줄 브랜드 스테이트먼트**: "10년차 백엔드 엔지니어 | 대규모 시스템 설계 전문" 형태
- **핵심 수치 3개**: "경력 N년 | 프로젝트 N개 | 오픈소스 기여 N건" — 숫자는 신뢰감을 줌
- **현재 소속/직함 명시**: 프로필 영역에서 바로 보이도록

```
현재:  [사진] [이름] [연락처] [공지]
제안:  [사진] [이름 + 한줄 소개] [핵심 수치 3개] [연락처] [CTA 버튼]
```

#### 전략 2: "초두 효과(Primacy Effect)" — 섹션 순서 재배치

사람은 처음 본 정보를 더 잘 기억합니다. 현재 순서는 Introduce(장문) → Skill → Experience인데, 채용담당자가 가장 먼저 확인하고 싶은 정보 순서와 맞지 않습니다.

**제안 섹션 순서:**

| 순서 | 섹션 | 근거 |
|------|------|------|
| 1 | **Profile (강화)** | 3초 안에 핵심 파악 |
| 2 | **Highlight (신규)** | 핵심 성과 3~5개 — "이 사람이 뭘 할 수 있는지" 즉시 전달 |
| 3 | **Experience** | 경력 상세 — 성과 중심으로 재구성 |
| 4 | **Project** | 구체적 프로젝트 성과 |
| 5 | **Skill** | 기술 스택 (경력 확인 후 검증 용도) |
| 6 | **OpenSource** | 차별화 포인트 — 실력의 외부 증거 |
| 7 | **Presentation** | 커뮤니티 영향력 |
| 8 | **Article/Award** | 외부 인정 |
| 9 | **Education** | 보조 정보 |
| 10 | **Introduce** | 개인적 스토리 — 관심 있는 사람만 읽음 |
| 11 | **Footer** | 유지 |

#### 전략 3: "사회적 증거(Social Proof)" — 추천/평가 섹션 추가

**채용담당자의 심리**: "본인이 쓴 건 다 좋게 쓰지. 남들은 뭐라고 하지?"

**제안**: `Testimonial` 또는 `Recommendation` 섹션 신규 추가
- 전/현 동료, 상사, 멘티의 한 줄 추천사
- LinkedIn 추천과 연동하거나 직접 수집한 인용문
- 추천인의 이름, 직함, 관계 명시 (신뢰도 향상)

#### 전략 4: "구체성 편향(Concreteness Bias)" — 성과를 숫자로

**채용담당자의 심리**: "되게 좋다" < "성능 30% 개선" — 구체적 숫자가 있을 때 신뢰도 급상승

**제안**: Experience/Project 설명을 STAR(Situation-Task-Action-Result) 또는 CAR(Challenge-Action-Result) 프레임워크로 재구성

```
❌ "Steering Kernel patching and updates for various distributions"
✅ "5개 주요 Linux 배포판에 대한 커널 패치 프로세스를 리드하여, 보안 취약점 대응 시간을 72시간 → 24시간으로 단축"
```

**description 데이터에 권장 패턴:**
- 정량적 성과: "N% 개선", "N건 처리", "N명 대상"
- 기술적 도전: 해결한 문제의 규모와 복잡도
- 비즈니스 임팩트: 매출, 비용 절감, 사용자 수 등

#### 전략 5: "희소성 원리(Scarcity Principle)" — 차별화 섹션 강조

**채용담당자의 심리**: "다른 지원자와 뭐가 다르지?"

오픈소스 기여, 발표 경험, 수상 이력은 대부분의 개발자가 갖지 못하는 **희소 자산**입니다. 하지만 현재는 페이지 하단에 매몰되어 있습니다.

**제안:**
- OpenSource/Presentation 섹션에 시각적 강조(아이콘, 색상 하이라이트) 추가
- Profile 영역의 핵심 수치에 오픈소스 기여 수 반영
- GitHub Stars, npm downloads 같은 외부 지표를 더 적극 활용

## 3. 스토리텔링 & 브랜딩 관점 개선 포인트

### 3.1 개인 브랜드 아이덴티티 구축

현재 이력서는 "정보의 나열"입니다. 브랜드로 전환하려면 **일관된 내러티브**가 필요합니다.

#### 브랜드 피라미드 프레임워크

```
        [존재 이유]
       "왜 이 일을 하는가?"

      [핵심 역량/가치]
     "남들과 다른 나만의 강점"

    [증거/실적]
   경력, 프로젝트, 오픈소스, 발표

  [기술 스택/도구]
 보유 기술, 사용 도구
```

**현재**: 피라미드 하단(기술/실적)만 나열
**개선**: 상단(존재 이유, 핵심 가치)부터 시작하는 서사 구조

#### 구체적 적용 방법

1. **브랜드 태그라인** (Profile에 추가)
   - 나만의 1줄 정의: "복잡한 시스템을 단순하게 만드는 엔지니어"
   - 이것이 전체 이력서의 **서사적 앵커(narrative anchor)** 역할

2. **Introduce 섹션 재구성** — 장문 에세이 → 구조화된 스토리
   - **Why**: 왜 이 분야를 선택했는가 (1~2문장)
   - **What**: 주로 어떤 문제를 풀어왔는가 (1~2문장)
   - **How**: 어떤 방식/철학으로 일하는가 (1~2문장)
   - **Where**: 앞으로 어디를 향하는가 (1~2문장)

3. **Experience의 서사 연결**
   - 각 경력이 독립적 항목이 아니라, 성장 과정의 챕터로 읽히도록
   - "인턴 → 정규직 → 리드" 같은 성장 곡선이 자연스럽게 보여야 함

### 3.2 시각적 브랜딩

| 요소 | 현재 | 제안 |
|------|------|------|
| 색상 체계 | 파란색(`#3c78d8`) 단일 | 주색 + 보조색 2개의 일관된 팔레트 |
| 타이포그래피 | Pretendard 단일 (weight 300) | 제목용 + 본문용 폰트 조합, weight 계층화 |
| 섹션 구분 | `<hr>` + 여백만 | 미묘한 배경색 교차, 또는 좌측 컬러 바 |
| 아이콘 활용 | 연락처에만 FontAwesome | 각 섹션/스킬 카테고리에 맥락적 아이콘 |
| 프로필 사진 | 사각형 이미지, 최대 320px | 원형 또는 부드러운 모서리, 배경 효과 |

### 3.3 "About Me" 마이크로카피 전략

이력서의 작은 텍스트들도 브랜딩 기회입니다:

- **섹션 제목**: `EXPERIENCE` → `Where I've Grown` (또는 커스텀 가능하게)
- **Skill 레벨 설명**: `1: 기초 수준` → `학습 중 📚` / `실무 적용 💼` / `전문가 🎯` (현재는 tooltip으로만 존재)
- **Footer**: 버전 번호만 → "Built with Next.js | 이 이력서도 제 작품입니다"

## 4. 차별화 인터랙션/기능 제안

### 4.1 Priority 1 — 높은 임팩트, 낮은 노력

| 기능 | 설명 | 기대 효과 |
|------|------|-----------|
| **플로팅 네비게이션** | 우측에 섹션 목차 플로팅, 현재 위치 하이라이트 | 긴 페이지에서의 탐색성 개선, 전문적 인상 |
| **핵심 성과 카드 (Highlight)** | Profile 아래에 3~5개 핵심 성과를 카드 형태로 | 3초 안에 핵심 역량 전달 |
| **스크롤 기반 섹션 진입 애니메이션** | 각 섹션이 뷰포트에 들어올 때 fade-in | 정적 페이지의 지루함 해소, 기술적 감각 어필 |
| **프로필 영역 CTA 버튼** | "이메일 보내기" / "이력서 다운로드" 버튼 | 채용담당자의 즉시 행동 유도 |
| **PDF/인쇄 최적화** | `@media print` CSS 또는 PDF 생성 기능 | 채용 프로세스 편의성 |

### 4.2 Priority 2 — 높은 임팩트, 중간 노력

| 기능 | 설명 | 기대 효과 |
|------|------|-----------|
| **다크 모드 토글** | 개발자답게 다크 모드 지원 | 기술 감각 어필, 사용자 편의성 |
| **스킬 시각화** | 레벨을 프로그레스 바 또는 도트 차트로 표현 | 직관적 역량 파악 |
| **타임라인 뷰** | Experience를 세로 타임라인으로 시각화 | 경력 흐름이 한눈에 보이는 스토리텔링 |
| **다국어 전환** | 한국어 ↔ 영어 토글 | 글로벌 채용 대응 |
| **추천사(Testimonial) 섹션** | 동료/상사 추천 인용문 캐러셀 | 사회적 증거 제공 |

| 기능 | 설명 | 기대 효과 |
|------|------|-----------|

## 5. SEO 및 소셜미디어 공유 최적화 전략

### 5.1 검색 엔진 최적화 (SEO)

#### A. 기술적 SEO

| 항목 | 현재 상태 | 개선안 |
|------|-----------|--------|
| **JSON-LD 구조화 데이터** | 없음 | `Person` + `ProfilePage` 스키마 추가 |
| **시맨틱 HTML** | `<div>` 위주 | `<header>`, `<main>`, `<section>`, `<article>`, `<nav>`, `<footer>` 적용 |
| **meta description** | "This is a simple web resume sample" | 개인화된 설명: "[이름] - [직함] | [핵심 역량] | [연락처]" |
| **canonical URL** | 미설정 | `<link rel="canonical">` 추가 |
| **sitemap.xml** | 없음 | 정적 사이트이므로 단일 URL sitemap 생성 |
| **robots.txt** | 없음 | 기본 허용 robots.txt 추가 |

#### B. JSON-LD 구조화 데이터 권장 스키마

```json
{
  "@context": "https://schema.org",
  "@type": "ProfilePage",
  "mainEntity": {
    "@type": "Person",
    "name": "이름",
    "jobTitle": "직함",
    "worksFor": {
      "@type": "Organization",
      "name": "회사명"
    },
    "url": "이력서 URL",
    "sameAs": ["GitHub URL", "LinkedIn URL"],
    "knowsAbout": ["기술1", "기술2", "기술3"]
  }
}
```

이를 통해 Google 검색 결과에서 이름 검색 시 **리치 스니펫(Knowledge Panel 형태)**이 표시될 수 있습니다.

#### C. 콘텐츠 SEO

- **페이지 타이틀**: `[이름] | [직함] - Resume` 형태로 최적화
- **헤딩 태그 계층**: 현재 `<h1>`(이름) → `<h2>`(섹션) 구조 유지하되, `<h3>` 이하 세분화
- **키워드 전략**: 채용담당자가 검색할 법한 키워드를 자연스럽게 포함
  - "[이름] 이력서", "[이름] 개발자", "[기술스택] 엔지니어" 등
- **이미지 alt 태그**: 프로필 사진, 뱃지 이미지에 의미 있는 alt 텍스트

### 5.2 소셜미디어 공유 최적화

#### A. OpenGraph 메타태그 강화

현재 `_global.ts`에 기본 OpenGraph가 설정되어 있지만 개선이 필요합니다:

| 항목 | 현재 | 개선안 |
|------|------|--------|
| `og:title` | "RESUME-NEXTJS Sample Page" | "[이름] - [직함] Resume" |
| `og:description` | "This is a simple web resume sample" | 핵심 역량을 포함한 매력적 설명 |
| `og:image` | `preview.jpg` (일반 이미지) | 이름 + 직함 + 핵심 스킬이 포함된 **OG 전용 카드 이미지** (1200x630px) |
| `og:type` | `profile` (적절) | 유지 |
| `profile:first_name` | "Lorem" | 실제 이름 |

#### B. 플랫폼별 공유 전략

| 플랫폼 | 전략 |
|--------|------|
| **LinkedIn** | 이력서 URL + "최근 이력서를 업데이트했습니다" 포스트. 프로필 URL 란에도 등록 |
| **GitHub** | README.md에 이력서 링크, pinned repository로 resume-nextjs 고정 |
| **Twitter/X** | `twitter:card=summary_large_image` 메타태그 추가, 프로필 bio에 URL |
| **개발자 커뮤니티** | 이력서 오픈소스 프로젝트 자체를 기술 블로그 포스트로 작성 |

#### C. OG 이미지 자동 생성 전략

정적 이미지 대신, 빌드 시 자동으로 OG 카드를 생성하는 방안:

- **도구**: `@vercel/og` 또는 `satori` 라이브러리
- **포함 내용**: 이름, 직함, 프로필 사진, 핵심 기술 3~5개
- **장점**: payload 변경 시 OG 이미지도 자동 업데이트

## 9. 핵심 가정 및 검증 필요 사항

| 가정 | 검증 방법 | 신뢰도 |
|------|-----------|--------|
| 채용담당자는 이력서를 6~7초 스캔한다 | Eye-tracking 연구 문헌 (Ladders, 2018) | HIGH |
| 성과 중심 서술이 업무 나열보다 효과적이다 | A/B 테스트 또는 채용담당자 인터뷰 | MEDIUM |
| 다크 모드가 개발자 이력서의 기술 감각을 높인다 | 사용자 피드백 수집 필요 | LOW |
| 플로팅 네비게이션이 체류 시간을 늘린다 | GA4 스크롤 깊이 비교 (전/후) | MEDIUM |
| JSON-LD가 실제 리치 스니펫 노출로 이어진다 | Google Search Console 모니터링 | MEDIUM |
| OG 이미지 최적화가 소셜 공유율을 높인다 | 공유 전/후 클릭률 비교 | MEDIUM |

## 부록: 참고 멘탈 모델

| 멘탈 모델 | 적용 포인트 |
|-----------|-------------|
| **초두 효과 (Primacy Effect)** | 섹션 순서 재배치 — 가장 중요한 정보를 맨 앞에 |
| **사회적 증거 (Social Proof)** | 추천사, GitHub Stars, 외부 뱃지 활용 |
| **희소성 원리 (Scarcity)** | 오픈소스/발표 등 희소 자산 강조 |
| **구체성 편향 (Concreteness Bias)** | 성과를 숫자와 구체적 결과로 표현 |
| **인지 용이성 (Cognitive Ease)** | 스캔 가능한 레이아웃, 시각적 계층 구조 |
| **Peak-End Rule** | 마지막 인상도 중요 — Footer를 의미 있게 |
| **Halo Effect** | 프로필 영역의 강한 첫인상이 나머지 평가에 영향 |
